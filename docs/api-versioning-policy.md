# API バージョニングと後方互換ポリシー

## バージョン管理方針

### URL バージョニング

本APIはURLパスにメジャーバージョンを含む形式を採用します。

```
https://api.example.com/api/v1/idols
https://api.example.com/api/v2/idols
```

現在のバージョン: **v1**

---

## 後方互換性の定義

### 後方互換な変更（破壊的変更ではない）

以下の変更は後方互換とみなし、マイナーバージョンアップまたはパッチとして扱います：

- レスポンスボディへのフィールド追加
- 新しいエンドポイントの追加
- 既存フィールドへのオプショナルなクエリパラメータ追加
- エラーメッセージの文言変更（`code` フィールドは変更しない）
- HTTP ヘッダーの追加

### 破壊的変更（メジャーバージョンアップが必要）

以下の変更は破壊的変更とみなし、新バージョン（v2等）として提供します：

- 既存フィールドの削除または名前変更
- フィールドの型変更（例: `string` → `integer`）
- 必須パラメータの追加
- エンドポイントパスの変更
- HTTPメソッドの変更
- レスポンスの構造変更（ネスト構造の変更など）
- 認証方式の変更（既存方式の廃止）
- エラー `code` の変更または削除

## バージョンアップ手順

### 新バージョン導入フロー

```
1. 新バージョン実装（/api/v2/...）
   ↓
2. Deprecation告知（最低90日前）
3. 旧バージョンにDeprecationヘッダーを付与
   ↓
4. 廃止日到達 → 旧バージョンを410 Goneで応答
   ↓
5. コードから旧バージョンを削除
```

### Deprecation ヘッダー

旧バージョンエンドポイントには以下のレスポンスヘッダーを付与します：

```
Deprecation: true
Sunset: Sat, 01 Jan 2027 00:00:00 GMT
Link: <https://api.example.com/api/v2/idols>; rel="successor-version"
```

## 破壊的変更の告知手順

1. **GitHub Issues** で破壊的変更の内容と移行手順を公開（廃止90日前）
2. **Deprecationヘッダー** をAPIレスポンスに付与
3. **READMEおよびSwagger** に廃止予定の旨を記載
4. **廃止日**以降は `410 Gone` を返す

### 410 Gone レスポンス例

```json
{
  "code": "VERSION_DEPRECATED",
  "message": "API v1 は廃止されました。v2 に移行してください。",
  "migration_guide": "https://github.com/kuro48/idol-database/wiki/v2-migration"
}
```

## バージョン切替ルール

### クライアント向けガイドライン

- バージョンはURLパスで指定し、`Accept-Version` ヘッダーは使用しない
- レスポンスに含まれる未知フィールドは無視する実装を推奨（後方互換変更への対応）
- エラー `code` フィールドに基づいてエラーハンドリングを実装する（HTTPステータスコードのみへの依存は避ける）

### 移行ガイドの提供

破壊的変更リリース時は以下を提供します：

- 変更点の詳細一覧
- v1 → v2 の差分対応表
- 移行スクリプトのサンプル（該当する場合）
- 新旧エンドポイントの並行稼働期間
